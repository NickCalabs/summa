# v0.2 Handoff Overview — Crypto & Wallet Tracking

> Read this first if you're picking up any v0.2 chunk.

## What you're working on

Summa is an open-source self-hosted Kubera replacement. Repo: `/opt/summa`. v0.1 is shipped — manual entry, ticker-based stocks/crypto, charts, multi-currency, Plaid, SimpleFIN, dashboard. The next milestone is **v0.2: first-class crypto wallet tracking**, parity with Kubera's on-chain support.

The v0.2 features are split into 5 chunks, each landed as one PR by a separate session. They are designed to be sequenced — BTC first, then ETH, then SOL, then DeFi/exchange, then settings. Each chunk's handoff is a standalone spec.

| Chunk | Handoff | Sequence |
|---|---|---|

| ETH balance + ERC-20 token detection (Etherscan) | `v0.2-eth.md` | Second |

## Shared assumptions across all v0.2 chunks

### Schema is already prepared

Look at `src/lib/db/schema.ts`. The relevant pieces:

- `assets.providerType` enum already includes `"wallet"` and `"exchange"`
- `assets.providerConfig` is JSONB and free-form per provider — the basic plan is `{ chain: "btc"|"eth"|"sol", address: "...", source: "blockstream"|"etherscan"|"helius" }`
- `assets.lastSyncedAt` is the per-asset sync timestamp
- `assets.staleDays` lets the UI mark a wallet as stale

**You should not need new tables for the basic single-address watch-only case.** A wallet asset is just an asset row with `providerType="wallet"`, the chain in `providerConfig`, and the value updated by a sync job.

If you decide a chunk needs new tables (e.g., per-token holdings as `holdings` table for nested expansion), document the migration in your PR.

### The cron pattern

`src/lib/cron.ts` is the source-of-truth for scheduled jobs. Existing jobs:

- Price refresh every 15 min (Yahoo + CoinGecko ticker assets)
- Daily snapshot at midnight UTC
- Plaid sync every 6h
- Exchange rate refresh daily

Add wallet sync as a new cron entry (suggested: every 30 min for BTC/ETH/SOL since on-chain data updates relatively slowly). Follow the existing pattern: query assets matching provider criteria, batch fetches, update each asset with the new value.

### The provider pattern

`src/lib/providers/` is where data adapters live. Existing adapters: `yahoo.ts`, `coingecko.ts`, `plaid.ts`, `simplefin.ts`, `exchange-rates.ts`. Each one is a self-contained module with typed inputs/outputs.

For each new chain, create a new provider file (`blockstream.ts`, `etherscan.ts`, `helius.ts`). Each should export:

- A function to fetch a single address's balance/holdings
- A function to fetch many addresses in one batch (if the API supports it)
- Internal rate-limit handling using `rate-limit-cache.ts`
- Typed errors (so cron can log + mark connection unhealthy)

### The optimistic-update pattern

When the user adds a wallet through the UI, follow the existing TanStack Query optimistic-update pattern in `src/hooks/use-assets.ts`. Use the `tempId()` helper from `@/lib/temp-id` (NOT `crypto.randomUUID()` directly — that breaks for self-hosters on plain HTTP, see `feedback_secure_context_apis.md`).

### Data layer check

Before writing any code, walk the relevant DB tables and confirm the schema supports what your UI needs. If the answer is "needs a new table" (e.g., `wallet_holdings` for token expansion), generate the migration via `pnpm db:generate` first, then apply with `pnpm db:migrate`, then restart the dev server (Drizzle caches schema metadata at startup — see `feedback_drizzle_restart.md`).

### Failure modes to plan for

For every chunk, address these in your plan:

- **Race conditions** — `await` ordering on multi-step flows (creating an asset, then adding sub-holdings, etc.)
- **Optimistic update rollback** — what happens when the API call fails after the optimistic insert
- **Stale providerConfig** — if the user edits a wallet address, ensure the next sync uses the new address, not the cached one
- **Rate limit exhaustion** — log + back off, don't retry tightly
- **Empty wallets** — should still create the asset row with value=0, not error
- **Address validation** — reject obviously invalid addresses at the API layer with a 400, not a 500 from the upstream provider

### Out of scope for every v0.2 chunk

- Sending transactions (read-only only)
- Importing private keys or seed phrases (we never touch private keys)
- Ledger / hardware wallet integration (later)
- xpub / hierarchical deterministic derivation (BTC chunk has a follow-up note about this)
- Historical backfill (we start tracking from "now"; no attempt to reconstruct past balances)
- Per-token snapshot history (in v0.2 we snapshot the wallet's total value, not each token; that's v0.4 territory)

## Working norms

- **Never commit directly to master.** Branch + PR + merge. See `AGENTS.md`.
- **Nick handles git commits himself unless he explicitly asks you to commit.**
- **Ship over polish.** Working features first, visual polish is timeboxed.
- **Don't duplicate the same data point in two places on one screen.**
- **Plans must explicitly address race conditions, optimistic update strategy, failure modes, and the data layer question.** "Should work" doesn't cut it.
- **Work sequentially through prioritized lists, not all-at-once.**
- **Test your changes by actually loading the app at `http://192.168.1.244:3000` and clicking through the flow** before claiming done.
