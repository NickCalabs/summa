# v0.2 Chunk 4 — Provider Settings Page + Connection Health Surface

> Read `v0.2-overview.md` first. This chunk depends on at least one wallet chunk (BTC/ETH/SOL) being shipped first. It also retroactively improves Plaid and SimpleFIN visibility.

## Goal

Give the user a single page where they can see and manage all their data-source connections — wallet addresses, Plaid links, SimpleFIN servers — with health indicators showing last sync, errors, and rate limit status.

This is what Kubera calls "Account Connections" and what makes the self-hosted experience feel polished. Without it, users add wallets and forget about them; with it, they have a control center.

## Scope

- New top-level page: `/settings/providers` (or `/settings/connections`)
- Sidebar nav item: "Connections" or "Data Sources"
- Sections on the page:
  1. **Wallets** — list of all `providerType="wallet"` assets, grouped by chain, with sync status
  2. **Plaid** — list of all Plaid connections (extends existing Plaid management UI)
  3. **SimpleFIN** — list of all SimpleFIN connections (extends existing SimpleFIN UI)
  4. **Price feeds** — read-only display of Yahoo + CoinGecko status
  5. **Exchange rates** — read-only display of frankfurter cache age
- For each connection: name, last synced, status (green/yellow/red), error message if any, manual sync button
- A global "Sync all" button that triggers all connections at once
- Settings: edit a wallet's name, regenerate connection, remove

## Out of scope

- Adding new connections from this page (those still go through the existing add-asset / add-connection flows)
- Per-asset sync history (separate v0.4 work)
- API key management for Yahoo/CoinGecko (they're configured via env vars; surface read-only status only)
- Webhook configuration UI
- Role-based access (single-user mode for v0.1-v0.2)

## Data layer

For wallet sync state:
- `assets.lastSyncedAt` already exists — use it
- `assets.metadata.lastSyncError` — add this if not present (free-form, no schema change needed)
- `assets.metadata.lastSyncStatus` — `"ok" | "error" | "stale"`

For Plaid:
- `plaid_connections.error_code`, `error_message`, `error_expires_at`, `last_synced_at` already exist (from migration 0004)

For SimpleFIN:
- `simplefin_connections.error_code`, `error_message`, `last_synced_at` already exist (from migration 0008)

So no new tables. Just normalize how each provider exposes "health" via a small adapter layer.

## API surface

### `GET /api/connections`

Returns a unified list across all provider types:

```ts
{
  wallets: [
    { id, name, chain, address, lastSyncedAt, status, error?, currentValueUsd }
  ],
  plaid: [
    { id, institutionName, lastSyncedAt, status, errorCode?, errorMessage?, accountCount }
  ],
  simplefin: [
    { id, label, serverUrl, lastSyncedAt, status, errorCode?, errorMessage?, accountCount }
  ],
  priceFeeds: [
    { name: "yahoo", lastFetchAt, status, error? },
    { name: "coingecko", lastFetchAt, status, error? },
    { name: "frankfurter", lastFetchAt, status, error? }
  ]
}
```

Compute `status` per connection:
- `"ok"` — last synced within expected window, no errors
- `"stale"` — last synced > 2x cron interval ago, no errors
- `"error"` — has an error_code or error_message
- `"never"` — never synced

### `POST /api/connections/sync` (sync all)

Triggers each provider's sync function in parallel. Streams or polls for completion.

### `POST /api/connections/wallet/[id]/sync`
### `POST /api/connections/plaid/[id]/sync` (extends existing)
### `POST /api/connections/simplefin/[id]/sync` (extends existing)

Per-connection manual sync triggers.

## Failure modes

| Mode | Handling |
|---|---|
| User triggers Sync all while another sync is running | Disable button while in flight; show spinner |
| One provider fails during Sync all | Continue with the others, show partial success toast |
| Plaid token expired | Surface a "Reconnect" CTA that triggers the existing reconnect flow |
| SimpleFIN rate-limited (24 req/day) | Show last-attempt time, disable button until next cycle |
| Wallet provider rate-limited | Same — surface the rate limit info from `rate-limit-cache.ts` |
| User removes a wallet | Cascade to delete the asset row (with confirmation listing how much value is being removed) |

## Race conditions

- Multiple tabs hitting Sync all simultaneously → use a per-user lock in the cron module (e.g., a Map keyed by userId, set on start, cleared on completion). If lock held, return "sync already in progress" 409.
- User clicks Sync now on one connection while Sync all is running → either queue or 409.
