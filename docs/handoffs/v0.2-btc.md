# v0.2 Chunk 1 — BTC Watch-Only Address Tracking

> Read `v0.2-overview.md` first.

## Goal

Let a user add a Bitcoin address as an asset row. The asset displays the current BTC balance and its USD-equivalent value, syncs every 30 minutes via a cron job, and updates the dashboard the same way ticker assets do.

This is the first crypto chunk because it's the simplest: BTC has the simplest address model, no token detection, no smart contracts, and the public APIs (Blockstream + Mempool.space) need no auth.

## Scope (this PR)

- Single watch-only address support (paste a `bc1...` / `1...` / `3...` address into a form, get a tracked asset)
- Provider: Blockstream Esplora API + Mempool.space as fallback
- BTC → USD conversion via the existing CoinGecko price feed (we already have the bitcoin price)
- Cron entry: every 30 minutes, refresh all `providerType=wallet` + `chain=btc` assets
- Address validation at the API layer (regex + length check, not full base58/bech32 validation)
- Empty wallet → asset row with `currentValue=0`, no error
- Rate-limit handling per `rate-limit-cache.ts` pattern

## Out of scope for this PR

- xpub / ypub / zpub support (extended public keys for HD wallets) — separate follow-up PR. Note in BACKLOG.md after this lands.
- Address book / labels — `assets.name` is enough
- Multiple addresses in one asset row (one address = one asset row in v0.2)
- Lightning channels
- Inscriptions / Ordinals balances
- Per-transaction history (covered in v0.4 transaction work)
- Tax lots / cost basis (v0.4 again)

## Data layer

Schema is already prepared. New BTC asset rows look like this:

```ts
{
  providerType: "wallet",
  type: "crypto",
  currency: "BTC",  // native currency
  quantity: "0.5",  // BTC amount
  currentPrice: "65000.00",  // USD per BTC at last sync
  currentValue: "32500.00",  // quantity * price, in BTC's currency (BTC for raw, USD if you store in USD)
  providerConfig: {
    chain: "btc",
    address: "bc1qxy2kgdygjrsqtzq2n0yrf2493p83kkfjhx0wlh",
    source: "blockstream"
  },
  lastSyncedAt: <timestamp>,
}
```

**Decision point:** store BTC value natively (currency=BTC, quantity=balance, price=BTC/USD rate, value=quantity*price in USD?) OR store in portfolio currency (currency=USD, value=converted)?

**Recommendation:** store with `currency="BTC"` and let the existing display layer convert via the exchange-rate cache. This matches how ticker crypto assets already work in `currentValue / currency` and avoids stamping today's exchange rate into the snapshot.

If that's not viable (e.g. the snapshot table needs USD), store the BTC quantity in `quantity`, the BTC/USD rate in `currentPrice`, and the converted USD in `currentValue`. The snapshot row's `valueInBase` already does the conversion at snapshot time.

## API surface

### New: `POST /api/wallets/btc`
Body: `{ address: string, name?: string, sectionId: string }`
Behavior:
1. Validate address format (regex)
2. Hit Blockstream `/address/{addr}` to confirm it resolves
3. Fetch current balance
4. Insert an asset row with `providerType="wallet"`, `providerConfig={chain:"btc", address, source:"blockstream"}`
5. Return the new asset

Or — alternative — extend the existing `POST /api/portfolios/[id]/assets` route to accept `providerType="wallet"` with chain-specific config. Probably cleaner because it reuses the existing optimistic-update hook (`useCreateAsset`).

**Recommendation:** extend the existing assets route. Less new surface area, less hook duplication.

### New: `POST /api/wallets/refresh` (manual sync trigger)
Optional. Lets the user click "Sync now" on a wallet asset. Calls the same code the cron runs.

## Provider implementation: `src/lib/providers/blockstream.ts`

```ts
export interface BtcAddressInfo {
  address: string;
  balanceSats: bigint;  // raw satoshis (BigInt because Number has precision issues over 2^53)
  balanceBtc: number;   // for convenience
}

export async function getBtcBalance(address: string): Promise<BtcAddressInfo>;
export async function getBtcBalanceBatch(addresses: string[]): Promise<Map<string, BtcAddressInfo>>;
```

- Endpoint: `https://blockstream.info/api/address/{addr}` (returns `{chain_stats, mempool_stats}`)
- Confirmed balance = `chain_stats.funded_txo_sum - chain_stats.spent_txo_sum`
- Mempool balance = same minus mempool stats (decide whether to include unconfirmed; recommendation: yes, label it consistently)
- Rate limit: Blockstream has no auth and no documented hard limit, but be polite — max 10 req/min per address group, batch by sleeping 100ms between calls
- Fallback: if Blockstream returns 5xx or times out, try Mempool.space (`https://mempool.space/api/address/{addr}`) which has the same response shape
- Use `rate-limit-cache.ts` for in-memory throttling

## Cron

In `src/lib/cron.ts`, add:

```ts
cron.schedule("*/30 * * * *", async () => {
  const wallets = await db.select().from(assets)
    .where(and(
      eq(assets.providerType, "wallet"),
      eq(assets.isArchived, false),
      sql`${assets.providerConfig}->>'chain' = 'btc'`
    ));

  if (wallets.length === 0) return;

  const addresses = wallets.map(w => (w.providerConfig as any).address);
  const balances = await getBtcBalanceBatch(addresses);
  const btcUsdPrice = await getCoinGeckoPrice("bitcoin", "usd");

  for (const wallet of wallets) {
    const info = balances.get((wallet.providerConfig as any).address);
    if (!info) continue;
    await db.update(assets).set({
      quantity: String(info.balanceBtc),
      currentPrice: String(btcUsdPrice),
      currentValue: String(info.balanceBtc * btcUsdPrice),
      lastSyncedAt: new Date(),
      updatedAt: new Date(),
    }).where(eq(assets.id, wallet.id));
  }

  console.log(`[cron] BTC wallet refresh: ${wallets.length} updated`);
});
```

(Adapt to whichever currency-storage decision you made above.)

## Failure modes

| Mode | Handling |
|---|---|
| Blockstream down (5xx) | Try Mempool.space; if both fail, log + skip this cycle, leave `lastSyncedAt` unchanged |
| Address typo / invalid format | API returns 400 at the create step before any insert |
| Address that doesn't exist on chain | Blockstream returns valid response with zero stats; treat as a real (empty) wallet |
| Rate limit hit | Backoff per `rate-limit-cache.ts`, retry on next cron tick |
| User edits asset row to a different address | Cron picks up the new `providerConfig.address` on next tick |
| Race: user creates and immediately syncs | The create flow already does an initial fetch, so the row has data on insert |
