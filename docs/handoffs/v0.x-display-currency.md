# v0.x Handoff — Display Currency (USD / BTC / sats)

> Read `v0.2-overview.md` for working norms (this isn't a v0.2 wallet chunk but the same norms apply).

## Goal

Let the user view the entire portfolio in a different display currency — USD, BTC, or sats — and have **everything** re-denominate including charts, the hero net worth, stats cards, the recap Sankey, the allocation donut, and individual asset rows. The chart for "net worth in BTC" must use **historical** BTC/USD rates per snapshot date so it shows real BTC purchasing power over time, not just a rescaled USD chart.

This is the Kubera display-currency feature. Reference: when Kubera is set to view in BTC, the net worth chart can become an almost-flat line for a BTC-heavy portfolio (you held BTC the whole time, so you "had" the same BTC) while the USD chart of the same portfolio fluctuates wildly with the BTC price.

Scope intentionally narrow for the first version: USD + BTC + sats only. The architecture is designed to extend to ETH, EUR, etc. in follow-up PRs.

## Why this is non-trivial

Three things make this harder than a simple "format with a different currency code":

1. **Charts need historical rates per snapshot date**, not today's rate. We currently store zero historical exchange rate data.
2. **The conversion layer is currently fiat-only** via frankfurter.app. Calling `convertCurrency` with `from="BTC"` falls through the graceful-fallback path and returns the raw number, so a 0.1 BTC asset currently contributes "$0.10" to USD net worth (a pre-existing bug masked because SimpleFIN imports always store USD). This bug must be fixed as part of this feature.
3. **Sats vs BTC** are different units of the same conversion target. Don't model them as two separate currencies in code; model sats as a display formatter on top of BTC.

## Scope (this PR)

- Display currency switcher in the top bar: `USD` / `BTC` / `sats`
- Persists to localStorage as `summa-display-currency`
- Re-denominates: hero net worth, stats cards, CAGR card, net worth chart, assets-vs-debts chart, recap Sankey, allocation donut/chart, sheet totals, section totals, asset row values, detail panel
- New `btc_usd_rate` column on `portfolio_snapshots`, populated by the daily snapshot cron from this PR forward
- Snapshot cron writes `btc_usd_rate` daily via CryptoCompare's free `/data/price` endpoint
- Crypto-aware conversion: extend `convertCurrency` to handle BTC alongside fiat (fixes the pre-existing 0.1 BTC = $0.10 bug)
- Dual-display on asset rows when the asset's native currency differs from the display currency
- Native unit storage works correctly for new ticker-flow crypto adds (small fix to ensure currency tagging is right)

**The BTC chart starts empty and populates over time.** Existing snapshot rows have `btc_usd_rate=NULL`; the chart simply skips those data points when in BTC/sats mode and renders only the dates where we have a stored rate. From the moment this PR ships, every new daily snapshot includes the rate, so the BTC chart populates one day at a time. The USD chart is unaffected — it still uses all existing history.

## Out of scope

- **Historical backfill of pre-existing snapshot dates** — see the "Optional historical backfill" section at the bottom for how to do this later if you want it. Not part of this PR.
- ETH, EUR, JPY, or any other display currency (architecture supports it; defer to a follow-up)
- Per-user DB-backed display preference (localStorage is enough for v0.x; future enhancement: `users.displayCurrency` column)
- Backfilling SimpleFIN imports to native crypto units (SimpleFIN doesn't expose raw crypto amounts — they sync as fiat balances)
- Cost basis / IRR in non-portfolio currencies (v0.4 analytics)
- Display currency in the Plaid / SimpleFIN sync flows themselves (the imports stay USD-stored; the display layer handles the rest)
- Migrating the snapshot cron's currentValue summation to be currency-aware (sum stays in portfolio currency; display layer re-denominates)
- A settings page for the display preference (it's a top-bar dropdown only for v0.x)

## Data layer

### Schema change

```sql
ALTER TABLE portfolio_snapshots
  ADD COLUMN btc_usd_rate numeric(20, 2);
```

Generate this via `pnpm db:generate`. Apply via `pnpm db:migrate`. **Restart the dev server after applying** — Drizzle caches schema metadata at startup (see `feedback_drizzle_restart.md`).

`btc_usd_rate` is nullable initially so existing rows don't break. The backfill script populates it. Going forward, the daily cron writes it on every new row.

### Snapshot cron change

In `src/lib/cron.ts`, the daily snapshot job currently computes per-portfolio totals and writes them to `portfolio_snapshots`. Add one line: fetch today's BTC/USD from CryptoCompare and include it in the insert.

```ts
async function fetchTodayBtcUsd(): Promise<number | null> {
  try {
    const res = await fetch(
      "https://min-api.cryptocompare.com/data/price?fsym=BTC&tsyms=USD"
    );
    if (!res.ok) return null;
    const data = await res.json();
    return data.USD ?? null;
  } catch {
    return null;
  }
}
```

In the snapshot loop:

```ts
const btcUsdRate = await fetchTodayBtcUsd();

await db.insert(portfolioSnapshots).values({
  // ...existing fields...
  btcUsdRate: btcUsdRate != null ? String(btcUsdRate) : null,
}).onConflictDoUpdate({
  // ...existing target...
  set: {
    // ...existing set fields...
    btcUsdRate: btcUsdRate != null ? String(btcUsdRate) : sql`btc_usd_rate`,
  },
});
```

If CryptoCompare is down for a day, `btcUsdRate` stays null for that row and the BTC chart will show a one-day gap. The next day's cron tick will work normally — no recovery logic needed.

## Crypto-aware conversion (the bug fix)

`src/lib/currency.ts` has `convertCurrency(amount, from, to, rates, rateMapBase)` which uses a `rates: Record<string, number>` map. Today, that map only contains fiat (29 entries from frankfurter, base USD).

**Fix:** extend the rates map at the boundary where it's constructed to include `BTC`. Two boundaries:

1. **API responses that return rates** (`/api/portfolios/[id]` returns the portfolio tree with `rates` and `ratesBase`). Server-side, after fetching the frankfurter rates, also fetch today's BTC/USD from CryptoCompare and merge: `rates.BTC = btcUsdRate`. The rate map base is USD, and the convention for `rates` is "1 base = X foreign", so for BTC where `1 USD = 1/65000 BTC`, the value is `1 / btcUsdRate`. **Get this math right** — see the `convertCurrency` function for how rates are interpreted (it currently does `amount / rate` for foreign→base).

2. **CurrencyContext provider** in `src/contexts/currency-context.tsx`. Already takes a rates map. No change needed if the API delivers the merged map.

```ts
// Server-side, in the portfolio tree handler:
const fiatRates = await getExchangeRates("USD"); // existing
const btcUsdRate = await getCurrentBtcUsd();      // new
const rates = { ...fiatRates };
if (btcUsdRate) {
  rates.BTC = 1 / btcUsdRate; // because 1 USD = 1/65000 BTC
}
return { ..., rates, ratesBase: "USD" };
```

Verify by writing a unit test: `convertCurrency(0.1, "BTC", "USD", { BTC: 1/65000 }, "USD")` should return `6500`. And `convertCurrency(6500, "USD", "BTC", { BTC: 1/65000 }, "USD")` should return `0.1`.

**Caveat:** the rate semantics in `convertCurrency` are "1 base = rate[X] of X", so for fiat rates from frankfurter (base USD) where `1 USD = 0.92 EUR`, the entry is `EUR: 0.92`. For BTC where `1 USD = 0.0000154 BTC`, the entry must be `BTC: 1/btcUsdRate`. Read the existing `convertCurrency` carefully and add unit tests before trusting any of this.

## Display layer

### New: `DisplayCurrencyContext`

Separate from the existing `CurrencyContext` to avoid confusion. `CurrencyContext` is "rates for the portfolio's base currency"; `DisplayCurrencyContext` is "what currency the user wants to view in".

```tsx
// src/contexts/display-currency-context.tsx
"use client";

import { createContext, useContext, useEffect, useState } from "react";

export type DisplayCurrency = "USD" | "BTC" | "sats";

interface DisplayCurrencyContextValue {
  displayCurrency: DisplayCurrency;
  setDisplayCurrency: (c: DisplayCurrency) => void;
  /** Convert a USD amount to the current display currency using today's rate */
  convertCurrent: (usdAmount: number, btcUsdRate: number) => number;
  /** Convert a USD amount to display currency using a specific historical rate */
  convertHistorical: (usdAmount: number, historicalBtcUsdRate: number) => number;
  /** Format a value (already in display currency) for display */
  format: (displayValue: number) => string;
}

const STORAGE_KEY = "summa-display-currency";
const Ctx = createContext<DisplayCurrencyContextValue | null>(null);

export function DisplayCurrencyProvider({ children }: { children: React.ReactNode }) {
  const [displayCurrency, setState] = useState<DisplayCurrency>("USD");

  useEffect(() => {
    try {
      const stored = localStorage.getItem(STORAGE_KEY) as DisplayCurrency | null;
      if (stored === "USD" || stored === "BTC" || stored === "sats") {
        setState(stored);
      }
    } catch {}
  }, []);

  const setDisplayCurrency = (c: DisplayCurrency) => {
    setState(c);
    try { localStorage.setItem(STORAGE_KEY, c); } catch {}
  };

  const convertCurrent = (usd: number, rate: number) => {
    if (displayCurrency === "USD") return usd;
    const btc = usd / rate;
    return displayCurrency === "sats" ? btc * 1e8 : btc;
  };

  const convertHistorical = convertCurrent; // same math, different rate input

  const format = (val: number) => {
    if (displayCurrency === "USD") {
      return new Intl.NumberFormat("en-US", { style: "currency", currency: "USD" }).format(val);
    }
    if (displayCurrency === "BTC") {
      return `\u20bf ${val.toFixed(val >= 1 ? 4 : 6)}`;
    }
    // sats
    return `${Math.round(val).toLocaleString("en-US")} sats`;
  };

  return (
    <Ctx.Provider value={{ displayCurrency, setDisplayCurrency, convertCurrent, convertHistorical, format }}>
      {children}
    </Ctx.Provider>
  );
}

export function useDisplayCurrency() {
  const v = useContext(Ctx);
  if (!v) throw new Error("useDisplayCurrency must be used inside DisplayCurrencyProvider");
  return v;
}
```

Wrap the app at `src/app/(app)/layout.tsx` (or wherever the `CurrencyProvider` is set up) so every authenticated screen has access.

### `DisplayCurrencyDropdown` — top bar

`src/components/portfolio/display-currency-dropdown.tsx`:

A small Base UI Select/DropdownMenu placed in `top-bar.tsx` next to the portfolio name. Three options. Persists via `useDisplayCurrency().setDisplayCurrency`. Active selection visible. Same Base UI gotcha applies — use `onClick` not `onSelect` (see commit `52d47df`).

### Asset row dual display

Where asset rows currently render their value via `MoneyDisplay`, replace with a small `AssetValueDisplay` that:

1. If `asset.currency === displayCurrency` → render the native amount only
2. Else → render the display-currency converted value as the main value, with the native amount as a smaller subtext below

```tsx
// Example for a 0.1 BTC asset, USD display:
// $6,500.00          ← main, large
// 0.1 BTC            ← subtext, smaller, muted

// Example for a USD bank account, BTC display:
// ₿ 0.193            ← main
// $12,500.00         ← subtext

// Example for a USD bank account, USD display:
// $12,500.00         ← just the main, no subtext
```

For SimpleFIN-imported assets where `quantity` is null, there's no native crypto unit to show — so when displayed in BTC mode, just show the BTC equivalent without subtext. When displayed in USD mode (their native currency), no subtext either. Subtext only shows when `asset.quantity != null` AND `asset.currency` is non-USD (or more generally, when the native and display differ AND the quantity is meaningful).

### Re-denomination at chart render time

Charts that read from `portfolio_snapshots`:

- `NetWorthChart` — currently maps each snapshot to `{ date, value: snapshot.netWorth }`. New: `{ date, value: convertHistorical(snapshot.netWorth, snapshot.btcUsdRate) }`. Y-axis tickFormatter swaps to use `format()`.
- `AssetsDebtsChart` — same pattern, applied to `totalAssets` and `totalDebts`.
- `RecapSankeyChart` — uses live portfolio data (today's view), not historical snapshots. Convert at render via `convertCurrent(usdValue, todayBtcUsdRate)`.
- `AllocationChart` — same: live data, convert at render.

For snapshot rows where `btc_usd_rate IS NULL` (gaps in the backfill), skip rendering that data point (the chart will visibly have a gap, which is honest about the missing data).

## Sats handling — important detail

Sats is **not a separate conversion currency**. It's a display formatter on top of BTC.

- The conversion math always goes USD → BTC. When `displayCurrency === "sats"`, the converted BTC is then multiplied by 1e8 in the format function.
- The chart shape is identical between BTC and sats modes — it's the same data, just labeled differently.
- The dropdown has 3 options for clarity, but only 2 underlying conversion targets in code (USD, BTC).
- Internally, the helper functions can branch on `displayCurrency === "sats"` to apply the multiplier at format time.
- 1 BTC = 100,000,000 sats. Don't store or compute in sats — always store/compute in BTC and format to sats at the very last step.

## Failure modes

| Mode | Handling |
|---|---|
| CryptoCompare API down during daily cron | Snapshot still writes (without `btc_usd_rate`); chart shows a one-day gap. Next tick recovers naturally. |
| User changes display currency mid-page-load | TanStack Query cache doesn't need invalidation — re-denomination is done at render time. Just trigger re-render via context update. |
| `convertCurrency` called with `from="BTC"` but rates map missing `BTC` | Falls through to graceful-fallback (returns raw amount) — same bug as before, but now narrowed because the API ALWAYS includes BTC in the rates map. Add a test to assert the rates map always has BTC. |
| Asset has `currency="BTC", quantity=null` (shouldn't happen but possible from edge cases) | Treat as fiat-style: use `currentValue` directly. Don't try to back-compute quantity. |
| User on plain HTTP (non-localhost) tries to switch currency | localStorage works on HTTP, no Web Crypto API needed — should work fine. Verify. |
| Snapshot row has `btc_usd_rate=NULL` and chart needs to render | Skip that data point (honest gap) rather than substituting today's rate (would lie about historical accuracy). All pre-existing snapshot rows have NULL until/unless backfill is performed; chart in BTC mode starts visibly empty and populates over time. |
| User manually adds BTC via the UI but the conversion bug isn't fixed yet | They'd see "$0.10" net worth contribution. Make sure the bug fix lands BEFORE the dropdown ships, otherwise users get bitten as soon as they switch to BTC view. |

## Race conditions

- **Display currency change while a chart is mid-fetch**: TanStack Query returns the same data; the chart re-renders with the new display function. No race because the rates are stored in the snapshot rows, not the cache.
- **Two browser tabs with different display currencies**: each tab's state is independent in React; localStorage syncs the value but not the React state until the next mount. Acceptable. (Future: storage event listener for cross-tab sync — out of scope.)

## Optional: historical backfill (deferred)

If at some point the user wants the BTC chart to extend backward into history (rather than starting empty and populating from this PR forward), the data exists in two places:

- **`https://github.com/NickCalabs/bitflation`** — Nick's other repo. `bitflation/scripts/prepare-btc-data.mjs` fetches BTC daily prices from **CryptoCompare** (`https://min-api.cryptocompare.com/data/v2/histoday?fsym=BTC&tsym=USD&limit=2000&toTs=...`), free, no API key, paginated in chunks of 2000 days. Date range: **2013-04-28 to today**. Output is `bitflation/src/data/usd/btc-daily.json` — array of `{date, price}` sorted by date. The script has built-in spot-check validation against known historical events (2013 first $1k, 2017 ATH, 2021 peak, 2024 ATH).
- **Kubera CSV exports** (when the `v0.1.x-kubera-import.md` handoff is implemented) — may include per-date BTC rates per row. Cross-validation source.

If/when you want to do this:

1. Copy `bitflation/src/data/usd/btc-daily.json` into Summa as `data/btc-daily-usd.json` at the repo root (NOT inside `src/`, to keep it out of the client bundle)
2. Write `src/lib/historical-btc.ts` with a Map loader and `getHistoricalBtcUsd(date)` function (in-memory map, O(1) lookup)
3. Write `scripts/backfill-btc-rates.ts` that walks all `portfolio_snapshots` rows where `btc_usd_rate IS NULL`, looks up the date in the loader, and updates the row. Idempotent (only updates NULL rows).
4. Add `pnpm db:backfill-btc-rates` to package.json
5. Run it once. Verify spot checks against bitflation's known reference dates.

This is a few hours of work and not required for the v0.x feature to ship.
