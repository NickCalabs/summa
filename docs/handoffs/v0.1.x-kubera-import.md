# v0.1.x Handoff — Kubera Historical CSV Import

> Read `v0.2-overview.md` for working norms (it's labeled v0.2 but the norms apply to all handoffs).

## Goal

Let the user upload years of historical Kubera data, match each Kubera account to an existing Summa asset, and backfill historical snapshots so the net worth chart shows the full multi-year history. This is the difference between "I started using Summa today" and "Summa knows my entire financial history".

This is **NOT** the same as the existing CSV import (`csv-import-dialog.tsx` + `import-csv` route), which handles current-state flat data. This is historical, multi-row, account-matched, schema-aware import.

## Why this matters

Nick (the Summa creator + primary user) is migrating from Kubera. He has years of history in a Kubera CSV export. Without this, his Summa instance starts at "today's net worth" with no chart history — which makes the dashboard charts useless and breaks the value prop ("the balance sheet you actually own"). This is a strong v0.1.x close-out item, important enough that it could justify holding `v0.1.0` tagging.

## Scope

- New page or modal: `/import/kubera`
- Step 1 — Upload: drag-drop a Kubera CSV export
- Step 2 — Parse: detect Kubera's specific column format and show a preview
- Step 3 — Account matching: for each Kubera account in the CSV, present a dropdown to map to an existing Summa asset OR create a new one
- Step 4 — Confirm: show the proposed snapshot insert count and date range, then commit
- Step 5 — Result: success summary with link to dashboard

## Out of scope

- Generic CSV import for non-Kubera formats (existing csv-import-dialog handles other use cases; keep that separate)
- Automated re-import (this is one-shot per source)
- Import from PDF, screenshots, or scraped data (v0.7 AI features)
- Importing transactions (v0.4)
- Importing notes/tags/categories from Kubera (best-effort if simple, otherwise skip)
- Reverse-engineering Kubera's full data model (we only need balances per account per date)

## Data layer

The schema mostly already supports this. You'll be writing rows to:
- `assets` — one per matched/created Kubera account (if not already mapped to existing Summa asset)
- `asset_snapshots` — one per asset per date in the CSV
- `portfolio_snapshots` — recomputed from the above; you may need to write these directly if perf matters

**Critical:** `asset_snapshots.uniqueAssetDate` index prevents duplicate (assetId, date) rows. Use `INSERT ... ON CONFLICT DO UPDATE` to handle re-runs gracefully.

The `valueInBase` column on `asset_snapshots` needs the historical FX rate for that date. For v0.1.x scope, **assume USD-only history** OR use today's FX rate as a stand-in (and document that this is a known limitation). Real historical FX is a v0.4 problem (frankfurter.app supports historical, but it's a separate provider call per date).

## Kubera CSV format research needed

Before writing parser code, **get an actual Kubera CSV from Nick** and reverse-engineer the column layout. Don't guess. Likely fields:
- Account name
- Account type (asset / debt / cash / investment)
- Currency
- Per-date balance columns OR one row per (account, date) pair

Document the format in this handoff before coding.

## Account matching UI

This is the hard part. Existing Summa assets need to be matched to Kubera accounts. Options:

1. **Auto-match by name** — fuzzy match Kubera account name against Summa asset names. Show confidence score. User accepts/rejects each match.
2. **Manual dropdown** — for each Kubera account, dropdown showing all Summa assets + "Create new" option. No auto-suggest.
3. **Both** — auto-match with manual override.

**Recommendation:** Option 3. Auto-match by exact name first, then case-insensitive contains, then leave unmatched as "Create new". Show all suggestions in a table, let user adjust.

## API surface

### `POST /api/import/kubera/parse`
Body: multipart upload of CSV file
Returns: parsed structure (preview), no DB writes

```ts
{
  rowCount: number,
  dateRange: { from: string, to: string },
  accounts: [
    { kuberaName: string, type: string, currency: string, snapshotCount: number }
  ]
}
```

### `POST /api/import/kubera/commit`
Body: `{ matches: [{ kuberaAccountName: string, summaAssetId: string | "new", newAssetConfig?: {...} }], dryRun?: boolean }`
Returns: insert summary

```ts
{
  assetsCreated: number,
  snapshotsInserted: number,
  snapshotsUpdated: number,
  errors: string[]
}
```

`dryRun: true` should compute the deltas without writing — use this to give the user a confirmation screen with "We will create 3 new assets and insert 1,247 snapshots covering 2018-01-15 to 2026-04-09. Confirm?"

## Failure modes

| Mode | Handling |
|---|---|
| Malformed CSV | Parse error returned with line number, no DB writes |
| Duplicate (assetId, date) | INSERT ON CONFLICT DO UPDATE — overwrites |
| Account name with no match and user didn't pick "create new" | Skip with warning in result |
| Currency mismatch (Kubera EUR account → Summa USD asset) | Convert at today's rate (document the limitation) OR error and ask user to fix |
| Very large CSV (years × dozens of accounts × thousands of rows) | Stream-parse, batch insert in chunks of 1000, surface progress to UI |
| Snapshot for a date BEFORE the asset's `createdAt` | That's fine; `asset_snapshots` doesn't have a constraint on this |
| `portfolio_snapshots` aggregates need to be recomputed for every imported date | Either run the snapshot cron retroactively for each date, or write a one-shot script that recomputes from `asset_snapshots` |
